## 1. Introduction

GitHub Activity Insight는 GitHub 활동 데이터를 기반으로 개발자의 역량을 분석하고, 해석 가능한 피드백을 제공하는 웹 시스템이다. 본 문서는 시스템의 분석(Analysis) 단계 산출물로서, 핵심 유스케이스, 도메인 클래스, 사용자 인터페이스 초안을 정의한다. Conceptualization 단계에서 정의한 문제의식(정량 통계만으로는 역량 해석이 어려움)을 바탕으로, 본 단계에서는 기능 요구를 사용자 관점과 시스템 관점에서 구체화하여 이후 설계 및 구현 단계의 기준선을 마련한다.

프로젝트의 주요 특징은 다음과 같다.

- 유용성: 단순 활동량이 아닌 활동성, 기술 다양성, 협업도, 지속성을 종합 분석해 실질적 개선 방향을 제시한다.
- 중요성: 취업 준비생과 주니어 개발자가 GitHub 포트폴리오를 객관적으로 점검하고 보완할 수 있는 근거를 제공한다.
- 확장성: GitHub API 연동 모듈, 분석 모듈, 피드백 엔진, 리포트 생성기를 분리한 구조로 기능 확장이 용이하다.
- 추적 가능성: 분석 결과를 저장하고 이력을 비교할 수 있어 개선 전후 변화 확인이 가능하다.

본 분석 단계의 범위(Scope)는 다음과 같다.

- In Scope: GitHub ID 입력부터 데이터 수집, 지표 계산, 점수/피드백 생성, 결과 조회/비교/PDF 출력까지의 기능 흐름 정의
- In Scope: 핵심 도메인 클래스와 클래스 간 관계 식별, 사용자 화면 구조 및 시나리오 정의
- In Scope: 보안 강화(글로벌 예외 처리, URL 인젝션 방어, 엔드포인트 인증)
- In Scope: PR/Issue 수집 기능 및 데이터 검증 메커니즘
- Out of Scope: 실제 알고리즘 파라미터 튜닝, 인프라 배포 아키텍처, 상세 DB 스키마 및 구현 코드

분석 단계 산출물의 목표는 기능 요구를 구조화하여 설계(Design) 단계에서 바로 상세 설계로 이어질 수 있도록 하는 데 있다.

**보안 및 인증 강화:**
- GitHub OAuth 2.0을 통한 안전한 사용자 인증(PKCE 플로우 적용)
- 글로벌 예외 처리 및 상세 로깅으로 보안 이벤트 추적
- URL 인젝션 방어 및 입력값 검증 강화
- 모든 분석 결과 조회 엔드포인트에 인증 및 권한 검증 적용
- 각 사용자별 독립된 세션 및 분석 이력 관리
- 개인 정보 보호 및 데이터 격리를 통한 접근 제어

### 3.2 Domain class description

[표 3-1] Domain class description

| 이름 | 설명 |
| :--- | :--- |
| User(사용자) | 시스템을 이용하는 인증된 사용자 클래스이다. GitHub OAuth 로그인을 통해 생성되며, 자신의 GitHub ID, 이메일, 프로필 정보, 세션 정보를 관리한다. 분석 요청과 결과 조회의 주체가 되며, 각 사용자는 독립된 분석 이력을 가진다. |
| AnalysisTarget(분석 대상) | 분석 대상이 되는 GitHub 계정을 나타내는 클래스이다. GitHub ID, 프로필 주소, 검증 여부를 속성으로 가지며 외부 활동 데이터 수집의 기준이 된다. User와 1:1 매핑되거나 다대다 매핑될 수 있다(다른 사용자의 GitHub ID도 분석 가능). |
| AnalysisSession(분석 세션) | 1회의 분석 실행을 나타내는 클래스이다. 요청 시각과 처리 상태를 관리하며, 데이터 수집부터 결과 생성까지의 분석 수명주기를 표현한다. User와 1:다 관계로, 각 사용자는 여러 분석 세션을 가질 수 있다. |
| ActivityEvidence(활동 근거) | GitHub로부터 수집한 원천 활동 데이터를 나타내는 클래스이다. 저장소, 커밋, 협업 활동 등 평가의 근거가 되는 사실 데이터를 포함한다. |
| EvaluationModel(평가 모델) | 수집된 활동 근거를 정량 지표로 해석하는 클래스이다. 활동성, 다양성, 협업도, 지속성 점수와 신뢰도 정보를 포함하며 결과 산출의 기준으로 사용된다. |
| InsightReport(인사이트 리포트) | 최종 분석 결과를 나타내는 클래스이다. 종합 점수, 개발자 유형, 생성 시각 등을 포함하며 결과 화면 표시와 이력 비교의 기준이 된다. |
| Feedback(피드백) | 분석 결과에 대한 해석 메시지를 나타내는 클래스이다. 강점, 약점, 개선 액션 등의 설명을 제공함으로써 사용자의 후속 행동을 지원한다. |

User(사용자)는 하나의 AnalysisTarget(분석 대상)을 등록하고, 이를 대상으로 복수의 AnalysisSession(분석 세션)을 생성할 수 있다. 각 AnalysisSession은 특정 AnalysisTarget에 대한 ActivityEvidence(활동 근거)를 수집하고, 수집된 데이터를 EvaluationModel(평가 모델)에 따라 해석하여 하나의 InsightReport(인사이트 리포트)를 생성한다. 또한 생성된 InsightReport에는 하나 이상의 Feedback(피드백)이 포함되며, 이를 통해 정량적 분석 결과가 사용자가 이해할 수 있는 개선 조언의 형태로 제공된다.

### 3.3 Domain constraints and business rules

| 규칙 ID | 제약/규칙 |
| :--- | :--- |
| BR-00 | User는 GitHub OAuth 2.0을 통해 인증되어야만 시스템 접근이 가능하며, 인증 후 생성된 세션 ID를 통해 추적된다. |
| BR-01 | 각 User의 sessionId는 유일해야 하며, 타임아웃(기본 30분)이나 명시적 로그아웃 시 무효화된다. |
| BR-02 | AnalysisTarget을 식별하는 GitHub ID는 공백이 아닌 영문/숫자/하이픈 조합이어야 한다. |
| BR-03 | AnalysisSession의 상태는 PENDING -> RUNNING -> COMPLETED/FAILED/PARTIAL 상태 전이 규칙을 따른다. (UC-03 확장 시나리오에서 PARTIAL 상태 발생 가능) |
| BR-04 | 동일 사용자의 RUNNING 상태 요청은 최대 1건으로 제한한다(중복 분석 방지). |
| BR-05 | InsightReport의 총점은 0~100 범위를 벗어날 수 없다. |
| BR-06 | EvaluationModel의 입력 근거가 임계치 미만인 경우 결과에 신뢰도 경고를 포함해야 한다. 구체적 임계치: 커밋 데이터 30개 미만, 활동 기간 3개월 미만, 저장소 1개 미만인 경우 신뢰도를 LOW로 표시. |
| BR-07 | 하나의 InsightReport는 반드시 1개의 EvaluationModel 해석 결과와 1개 이상의 Feedback을 포함해야 한다. |
| BR-08 | 사용자별 분석 이력은 InsightReport.createdAt 기준으로 누적 저장되어야 하며, UC-08 비교는 동일 User 소유 결과끼리만 허용한다. |
| BR-09 | AnalysisSession이 FAILED 또는 PARTIAL 상태로 종료된 경우, InsightReport에는 데이터 제한 또는 실패 사유가 추적 가능하게 기록되어야 한다. |

위 규칙은 설계 단계에서 상태머신 정의, 입력 검증 규칙, 데이터 무결성 제약으로 구체화한다.

## 5. Glossary

| 용어 | 정의 |
| :--- | :--- |
| GitHub API | GitHub가 제공하는 REST/GraphQL 인터페이스로 저장소, 커밋, 언어, 협업 활동 데이터를 조회하는 수단 |
| Raw Activity Data | API에서 수집한 원천 활동 데이터(저장소 목록, 커밋 로그, 언어 통계 등) |
| Metrics | 원천 데이터를 가공해 산출한 정량 지표(활동성, 다양성, 협업도, 지속성) |
| Competency Score | 다수 Metrics를 가중 결합해 계산한 종합 역량 점수 |
| Feedback | 점수/지표 해석을 통해 생성된 강점, 약점, 개선 액션 권고 |
| Analysis Request | 특정 시점의 분석 실행 단위 및 상태를 관리하는 요청 객체 |
| Analysis History | 사용자별로 누적된 AnalysisResult의 시간순 집합으로, 시점 간 비교를 위한 이력 데이터 |
| Report Generator | 결과를 대시보드 또는 PDF 형태로 변환/출력하는 모듈 |
| Rate Limit | GitHub API 호출 횟수 제한 정책으로, 초과 시 일정 시간 요청이 제한되는 제약 |
| Backoff Retry | API 오류/제한 발생 시 대기 시간을 점진적으로 늘려 재시도하는 전략 |
| Pagination | 대량 API 응답을 여러 페이지로 분할해 순차적으로 조회하는 방식 |
| ViewModel | 화면 출력 목적에 맞게 가공된 데이터 구조로 UI 렌더링 입력 모델 |
| TrustLevel | 분석 결과의 데이터 완전성 및 신뢰도를 나타내는 4단계 등급. HIGH(완전성 90% 이상) / PARTIAL(일부 데이터 누락) / LOW(주요 데이터 부족) / LIMITED(이상치 필터링 또는 임계값 미달) |
| Developer Type | 점수 범위에 따라 분류된 개발자 수준 카테고리. Beginner(0-39점), Junior(40-69점), Advanced(70-100점)로 구분 |
