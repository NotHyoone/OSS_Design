## 1. Introduction

### 1.1 주제 선정 배경

GitHub는 개발자의 프로젝트 경험, 협업 이력, 기술 스택 변화가 모두 축적되는 대표적인 개발 플랫폼이다. 취업 준비생과 주니어 개발자는 GitHub를 포트폴리오로 활용하지만, 기존 GitHub 통계 도구는 저장소 수, 커밋 수, 사용 언어 비율 등 단순 정량 정보만 제공할 뿐, 그 수치가 의미하는 강점과 약점 또는 개선 방향까지 설명하지 못한다. 프로젝트 개발자인 나 역시 "내 GitHub 활동이 실제로 어떤 역량으로 보이는지", "무엇을 어떻게 보완해야 하는지"를 파악하기 어려운 불편함을 직접 경험하였고, 이를 해소하기 위해 본 주제를 선정하였다.

### 1.2 문제 정의

본 프로젝트가 해결하고자 하는 핵심 문제는 다음과 같다.

- **역량 해석 불가:** 기존 통계 도구는 활동량 수치만 제공하며, 활동성·기술 다양성·협업도·지속성을 종합한 역량 해석을 제공하지 않는다.
- **개선 방향 부재:** 점수나 통계만으로는 어떤 부분을 어떻게 개선해야 하는지 구체적인 행동 지침을 도출하기 어렵다.
- **평가 불투명성:** 결과의 산출 근거가 명확히 제시되지 않으면 사용자가 결과를 신뢰하기 어렵고, 실질적인 개선 행동으로 연결되지 않는다.
- **이력 추적 불가:** 포트폴리오를 개선한 이후 어느 지표가 얼마나 향상되었는지 시계열로 비교할 수 있는 수단이 없다.

### 1.3 해결 접근 방식

GitHub Activity Insight는 위 문제를 다음 방식으로 해결한다.

- **데이터 수집:** GitHub REST API를 통해 저장소, 커밋 로그, 언어 통계, Pull Request·Issue 참여 이력 등 원천 활동 데이터를 수집한다.
- **지표 계산(Metrics):** 수집 데이터를 정규화하여 활동성, 기술 다양성, 협업도, 지속성의 4개 핵심 지표를 0–100 척도로 산출한다.
- **점수 및 피드백 생성:** 지표에 가중치를 적용하여 종합 역량 점수를 계산하고, Developer Type(Beginner/Junior/Advanced)을 분류한 뒤, 약점별 맞춤형 개선 액션 아이템을 생성한다.
- **결과 저장 및 이력 비교:** 분석 결과를 데이터베이스에 저장하여 재조회·이력 비교가 가능하게 하고, OpenPDF를 사용하여 A4 PDF 리포트를 서버에서 직접 생성하여 다운로드로 제공한다.

시스템은 웹 애플리케이션, GitHub API 연동 모듈, 분석·평가 엔진, 리포트 생성기, 결과 저장소로 구성되며, 각 모듈을 분리하여 지표 산출 규칙 변경 시 핵심 코드 수정 범위를 최소화한다.

### 1.4 주 타겟층

| 대상 | 활용 목적 |
| :--- | :--- |
| 취업 준비생 / 주니어 개발자 | GitHub 포트폴리오를 객관적으로 점검하고 개선 방향을 파악 |
| 개인 개발자 | 자신의 개발 이력과 협업 활동 패턴을 정리하고 성장 추이 확인 |
| 채용 담당자 / 교육 멘토 | 지원자·교육생의 GitHub 활동을 정량 근거와 함께 참고 자료로 활용 |

### 1.5 설계 단계의 목표

본 문서는 시스템의 설계(Design) 단계 산출물로서, Analysis 단계에서 도출한 유스케이스(UC-01~UC-08)와 도메인 모델을 구체적인 소프트웨어 구조로 전환한다. 설계 단계의 목표는 다음과 같다.

- 도메인 클래스 간 관계, 속성, 오퍼레이션을 클래스 다이어그램으로 명세하여 구현 청사진을 확립한다.
- 핵심 유스케이스 시나리오에 대한 시퀀스 다이어그램으로 객체 간 협력 흐름을 정의한다.
- AnalysisRequest 생명주기(PENDING → RUNNING → COMPLETED / FAILED)를 상태 머신 다이어그램으로 명세한다.
- 구현 시 준수해야 할 기술 제약 및 비기능 요구사항을 구현 요구사항으로 정리하여 개발자가 직접 참조할 기준선을 마련한다.

### 1.6 범위(Scope)

- **In Scope:** 클래스 다이어그램, 시퀀스 다이어그램, 상태 머신 다이어그램 작성, 구현 요구사항 및 기술 제약 정의, 글로벌 예외 처리 전략, URL 인젝션 방어, 엔드포인트 인증 강화
- **Out of Scope:** 실제 알고리즘 파라미터 수치 결정, 인프라 배포 아키텍처 상세 설계, 데이터베이스 물리 스키마 및 구현 코드 작성

본 문서의 구성은 다음과 같다. 2장에서는 시스템의 정적 구조를 표현하는 클래스 다이어그램을 제시하고(OAuth 및 User 관리 클래스 포함), 3장에서는 주요 유스케이스 시나리오에 대한 시퀀스 다이어그램으로 동적 흐름을 정의한다. 4장에서는 AnalysisRequest의 상태 전이를 상태 머신 다이어그램으로 명세하며, 5장에서는 구현 시 준수해야 할 기술 요구사항, 보안 제약, 글로벌 예외 처리 전략 및 제약 조건을 정리한다.

### 2.6 클래스 상세 설명

#### 2.6.1 도메인 엔티티 클래스

도메인 엔티티 클래스는 시스템의 핵심 데이터를 보관하며, 분석 흐름의 각 단계에서 생성·갱신·참조된다.

---

##### User

클래스 개요

| 항목 | 내용 |
| :--- | :--- |
| 역할 | GitHub OAuth 2.0을 통해 인증된 사용자 정보를 저장한다. 각 사용자의 세션, 분석 이력, 권한을 관리하는 중심 엔티티. |
| 참여 UC | UC-00 (GitHub OAuth 로그인), UC-01~08 (인증된 사용자 활동) |
| 관계 | User(1) → AnalysisRequest(0..*), User(1) → AnalysisResult(0..*) |

속성

| 이름 | 가시성 | 타입 | 기본값 | 설명 |
| :--- | :---: | :--- | :--- | :--- |
| userId | + | String | UUID.generate() | 시스템 내 사용자 고유 식별자. UUID 형식. |
| githubId | + | String | - | GitHub 사용자 식별자. 프로필 조회 및 데이터 수집의 기준. |
| email | + | String | - | GitHub 등록 이메일. 알림 및 연락 목적. |
| displayName | + | String | "" | GitHub 표시 이름. UI 렌더링용. |
| avatarUrl | + | String | "" | GitHub 아바타 이미지 URL. |
| sessionId | + | String | null | 현재 유효한 세션 ID. HttpOnly 쿠키에 저장됨. |
| createdAt | + | DateTime | now() | 계정 생성 시각. |
| lastLoginAt | + | DateTime | null | 마지막 로그인 시각. |

오퍼레이션

| 이름 | 시그니처 | 반환 타입 | 설명 |
| :--- | :--- | :---: | :--- |
| validate | validate() | Boolean | 필수 필드(userId, githubId, email)가 모두 채워져 있고 형식이 유효하면 true 반환. |
| isSessionValid | isSessionValid() | Boolean | sessionId가 null이 아니고 타임아웃이 경과하지 않았으면 true 반환. |
| updateLastLogin | updateLastLogin() | void | lastLoginAt을 현재 시각으로 갱신한다. |

**추가 설명**

- **세션 관리**: sessionId는 HttpOnly 쿠키에만 저장되며, JavaScript 접근 불가 (XSS 방어)
- **데이터 격리**: 사용자별 AnalysisResult 조회 시 userId 필터 적용으로 데이터 격리 확보
- **라이프사이클**: OAuth 콜백 후 생성 → 세션 활성 동안 유지 → 로그아웃 시 세션 무효화 → 일정 TTL 후 DB에서 삭제
- **입력 검증**: 모든 사용자 입력(userId, githubId 등)에 대해 null 체크, 길이 검증, 특수 문자 제거 수행
- **글로벌 예외 처리**: 인증 과정의 모든 단계에서 예외를 캡처하여 로깅하고 사용자에게 안전한 에러 메시지 제공

---

##### OAuthClient

클래스 개요

| 항목 | 내용 |
| :--- | :--- |
| 역할 | GitHub OAuth 2.0 API 호출을 캡슐화한다. Authorization Code + state 검증 기반 인증과 토큰 관리를 담당하는 외부 인증 클라이언트. |
| 참여 UC | UC-00 (GitHub OAuth 로그인) |
| 관계 | OAuthClient ..> AuthenticationService : «use» |

속성

| 이름 | 가시성 | 타입 | 기본값 | 설명 |
| :--- | :---: | :--- | :--- | :--- |
| clientId | - | String | - | GitHub OAuth App Client ID. 환경 변수로 주입. |
| clientSecret | - | String | - | GitHub OAuth App Client Secret. 환경 변수로 주입. 외부 노출 금지. |
| redirectUri | - | String | "https://localhost:8080/auth/callback" | OAuth 인증 완료 후 리다이렉트 URI. |
| authorizationUrl | - | String | "https://github.com/login/oauth/authorize" | GitHub OAuth 인증 엔드포인트. |
| tokenUrl | - | String | "https://github.com/login/oauth/access_token" | GitHub OAuth 토큰 엔드포인트. |

오퍼레이션

| 이름 | 시그니처 | 반환 타입 | 설명 |
| :--- | :--- | :---: | :--- |
| getAuthorizationUrl | getAuthorizationUrl(String state) | String | OAuth state를 포함한 GitHub 인증 URL을 반환한다. state는 CSRF 방어용이다. |
| exchangeCodeForToken | exchangeCodeForToken(String code) | String | 인증 코드를 사용해 Access Token을 획득하여 반환한다. |
| getUserProfile | getUserProfile(String accessToken) | Map\<String, Object\> | Access Token으로 GitHub API를 호출하여 사용자 프로필 정보를 반환한다. |
| isConfigured | isConfigured() | Boolean | OAuth Client ID/Secret 환경변수 설정 여부를 반환한다. |

**추가 설명**

- **인증 플로우**: 현재 구현은 Authorization Code Flow + state(CSRF 방어) 기반이다.
- **보안 고려사항**: clientSecret은 절대 클라이언트 측에 노출되지 않으며, 서버에서만 관리. 환경 변수로 안전하게 주입
- **상태 검증**: state 파라미터는 세션당 1회용이며, CSRF 공격 방어. 모든 redirectUri 파라미터에 대해 URL 인젝션 방어
- **입력 검증**: exchangeCodeForToken 호출 시 code 포맷 검증, 특수 문자 제거
- **URL 인젝션 방어**: getAuthorizationUrl 반환값에 대해 redirectUri와 기타 파라미터의 URL 인코딩 검증

**추가 설명**

- **상태 전이 규칙**: PENDING → RUNNING → (COMPLETED|PARTIAL|CANCELLED|FAILED). 역방향 전이는 허용되지 않음
- **재시도 전략**: FAILED 상태에서만 재시도 가능. 3회 초과 시 canRetry()=false로 최종 실패 처리

**추가 설명**

- **파이프라인 플로우**:
    1. requestAnalysis(): AnalysisRequest 생성 (DB 저장, 상태=PENDING)
    2. AnalysisAsyncRunner(@Async): executeAnalysis(req) 비동기 실행
    3. 각 단계에서 진행률(step/overallPct/detail) 갱신
    4. 실패 시 markError() 후 status=FAILED 저장
    5. 성공 시 markDone() 후 결과 저장
- **동시성 제어**: 동일 githubId에 대해 활성 요청(PENDING/RUNNING)이 있으면 기존 요청을 재사용한다.
- **재시도 처리**: retryFailedAnalysis()는 canRetry() 조건에서 retryCount 증가 후 비동기 재실행한다.

| saveResult | saveResult(AnalysisResult result) | void | AnalysisResult를 저장한다. |
| saveMetrics | saveMetrics(String requestId, Metrics metrics) | void | requestId 기준으로 Metrics를 저장한다. |
| findResultByRequestId | findResultByRequestId(String requestId) | AnalysisResult | requestId에 해당하는 AnalysisResult를 반환한다. 없을 경우 예외를 던진다. |
| findLatestResultByGithubId | findLatestResultByGithubId(String githubId) | Optional\<AnalysisResult\> | githubId 기준 최신 결과 1건을 조회한다. |
| findResultsByUserId | findResultsByUserId(String userId) | List\<AnalysisResult\> | 해당 userId의 모든 AnalysisResult를 createdAt 내림차순으로 반환한다. 이력 비교(UC-08)에 사용. |
| findResultsByGithubId | findResultsByGithubId(String githubId) | List\<AnalysisResult\> | githubId 기준 이력 목록을 최신순으로 반환한다. |
| findAllResults | findAllResults() | List\<AnalysisResult\> | 모든 AnalysisResult를 반환한다. (관리자·분석용) |
| findMetricsByRequestId | findMetricsByRequestId(String requestId) | Optional\<Metrics\> | requestId 기준 Metrics를 조회한다. |
| findActiveRequestId | findActiveRequestId(String githubId) | Optional\<String\> | githubId 기준 활성 요청(PENDING/RUNNING)의 requestId를 반환한다. |
| deleteExpired | deleteExpired(Integer ttlMinutes) | Integer | createdAt이 ttlMinutes분 이상 된 결과를 삭제하고 삭제 건수를 반환한다. (정기 정리용) |
| getStatistics | getStatistics() | Map\<String, Object\> | 전체 통계(총 분석 건수, 평균 점수, 타입 분포 등)를 반환한다. (대시보드용) |
| purgeExpired | purgeExpired() | void | 스케줄러(1시간 주기)로 오래된 request/result 및 멈춘 running 요청을 정리한다. |

**추가 설명**

- **데이터 격리**: findResultsByUserId() 호출 시 userId 파라미터로 필터링. 타 사용자 데이터 노출 방지
- **인덱싱 전략**:
  - AnalysisRequest: (userId, requestId) 복합 인덱스
  - AnalysisResult: (userId, createdAt DESC) 복합 인덱스 (이력 조회 최적화)
- **트랜잭션 관리**: save(request) 후 체계적으로 상태 갱신. 부분 실패 시 rollback
- **TTL 정책**: 스케줄러 purgeExpired()에서 요청 24시간, 멈춘 RUNNING 2시간, 결과 72시간 기준으로 정리한다.
- **배치 성능**: findByStatus()는 커서 기반 페이지네이션 지원 (대량 데이터 처리 시 메모리 절감)

### 3.2 SD-01: UC-00 GitHub OAuth 2.0 로그인

사용자가 [GitHub로 로그인] 버튼을 클릭한 시점부터 OAuth 인증을 완료하고 세션 쿠키가 발급되어 Home 화면으로 복귀하는 시점까지의 흐름이다. `AuthenticationService`는 state 값을 생성·검증하여 CSRF를 방어하고, `OAuthClient`를 통해 GitHub OAuth API와 토큰 교환 및 사용자 프로필 조회를 수행한다.

```mermaid
sequenceDiagram
    actor User as "User Browser"
    participant WebApp as WebApp
    participant AuthCtrl as AuthController
    participant AuthSvc as AuthenticationService
    participant OAuthCli as OAuthClient
    participant GitHub as GitHub OAuth API

    User->>WebApp: Click GitHub Login Button
    WebApp->>AuthCtrl: GET /auth/login
    AuthCtrl->>+AuthSvc: initiateOAuthFlow()
    AuthSvc->>AuthSvc: generateState()
    AuthSvc->>OAuthCli: getAuthorizationUrl(state)
    OAuthCli-->>AuthSvc: authorizationUrl
    AuthSvc-->>-AuthCtrl: authorizationUrl
    AuthCtrl-->>WebApp: 302 Redirect

    WebApp->>GitHub: Authorization Request(state)
    GitHub-->>User: Show Login / Consent Screen
    User->>GitHub: Authenticate and Authorize
    GitHub-->>WebApp: 302 Redirect /auth/callback?code&state

    WebApp->>AuthCtrl: GET /auth/callback
    AuthCtrl->>+AuthSvc: handleOAuthCallback(code, state)
    AuthSvc->>AuthSvc: validateState()
    AuthSvc->>+OAuthCli: exchangeCodeForToken(code)
    OAuthCli->>GitHub: POST /login/oauth/access_token
    GitHub-->>OAuthCli: access_token
    OAuthCli-->>-AuthSvc: accessToken
    AuthSvc->>+OAuthCli: getUserProfile(accessToken)
    OAuthCli->>GitHub: GET /user
    GitHub-->>OAuthCli: profileMap
    OAuthCli-->>-AuthSvc: GithubProfile
    AuthSvc->>AuthSvc: Create or Update User
    AuthSvc->>AuthSvc: createSession(user)
    AuthSvc-->>-AuthCtrl: authenticated User
```

## 4. State Machine Diagram

### 4.1 개요

본 장에서는 AnalysisRequest의 생명주기를 상태 머신 다이어그램으로 명세한다. 각 상태는 [2.6.1 AnalysisRequest](#analysisrequest)에서 정의한 RequestStatus 열거형 값에 대응하며, 상태 전이는 AnalysisService의 오퍼레이션(executeAnalysis, retryFailedAnalysis, cancel)에서 구현된다.

---

### 4.2 AnalysisRequest 상태 머신 다이어그램

```mermaid
stateDiagram-v2
    [*] --> PENDING: create(userId, githubId)

    PENDING --> RUNNING: executeAnalysis()

    RUNNING --> COMPLETED: Pipeline Success
    RUNNING --> FAILED: Error
    RUNNING --> CANCELLED: cancel(requestId)

    FAILED --> RUNNING: retryFailedAnalysis() [canRetry = true]
    FAILED --> [*]: Max Retries [canRetry = false]

    COMPLETED --> [*]
    CANCELLED --> [*]
```

### 4.3 상태 설명

[표 4-1] 상태 설명

| 상태 | RequestStatus | 설명 | 진입 조건 | 종료 조건 |
| :--- | :--- | :--- | :--- | :--- |
| 초기 | — | 객체 미생성 상태 | — | create() 호출 |
| 대기 | PENDING | 분석 요청 생성 완료, 큐 대기 중 | create() 성공, DB 저장 완료 | executeAnalysis() 호출 |
| 실행 중 | RUNNING | 데이터 수집·지표 계산·점수 생성 진행 중 | executeAnalysis() 시작 | 성공(COMPLETED) 또는 오류(FAILED) |
| 완료 | COMPLETED | 분석 파이프라인 전 단계 정상 완료, 신뢰도 HIGH | 모든 단계 성공 | 최종 상태 |
| 부분 완료 | PARTIAL | 열거형에 정의된 확장용 상태(현재 기본 파이프라인에서는 미사용) | 향후 부분 성공 경로 추가 시 | 최종 상태 |
| 취소 | CANCELLED | 사용자가 진행 중 분석을 취소한 상태 | cancel(requestId) 및 소유권 검증 성공 | 최종 상태 |
| 실패 | FAILED | 치명적 오류로 분석 불가 (retryCount < 3이면 재시도 가능) | GitHub API 오류, 타임아웃, 검증 실패 | retryFailedAnalysis() 또는 최종 실패 |
| 최종 실패 | FAILED | 재시도 횟수 초과, 더 이상 재시도 불가 | canRetry() = false (retryCount ≥ 3) | 최종 상태 |

---

### 4.4 상태 전이 규칙

[표 4-2] 상태 전이 규칙

| From | To | 트리거 | 가드 조건 | 수행 동작 |
| :--- | :--- | :--- | :--- | :--- |
| [초기] | PENDING | create(userId, githubId) | GithubProfile.validated = true | requestId 발급, requestedAt 기록, DB 저장 |
| PENDING | RUNNING | executeAnalysis() 비동기 실행 시작 | 동일 userId의 RUNNING 요청 없음 (BR-03) | status = RUNNING 갱신 |
| RUNNING | COMPLETED | 파이프라인 전 단계 성공 | AnalysisResult 저장 완료 | completedAt 기록, status = COMPLETED |
| RUNNING | CANCELLED | cancel(requestId) | 요청 소유자 또는 동일 GitHub ID 사용자 | errorMessage 기록, completedAt 기록, status = CANCELLED |
| RUNNING | FAILED | 오류 발생 | GitHub API 오류 / 타임아웃 / 검증 실패 | errorMessage 기록, completedAt 기록, status = FAILED |
| FAILED | RUNNING | retryFailedAnalysis() | canRetry() = true (retryCount < 3) | incrementRetry(), status = RUNNING |
| FAILED | [최종] | — | canRetry() = false (retryCount ≥ 3) | 최종 실패 확정, 더 이상 재시도 없음 |
| COMPLETED | [최종] | 결과 저장 완료 | — | — |
| PARTIAL | [최종] | 결과 저장 완료 | — | — |
| CANCELLED | [최종] | 사용자 취소 완료 | — | — |

**전이 제약 사항:**
- 역방향 전이 불가: COMPLETED/PARTIAL/CANCELLED → 이전 상태로의 전이 없음
- 동일 githubId의 PENDING 또는 RUNNING 상태 요청이 이미 존재하는 경우 신규 PENDING 생성 불가 (BR-03)
- PARTIAL 상태는 열거형에 정의되어 있으나 현재 executeAnalysis 기본 경로에서는 사용하지 않는다(향후 확장 reserved).

## 6. Glossary

[표 6-1] 설계 단계 용어집

| 용어 | 정의 |
| :--- | :--- |
| AnalysisRequest | 분석 요청의 생명주기를 추적하는 엔티티. PENDING → RUNNING → COMPLETED/FAILED 상태 전이 관리. |
| ActivityData | GitHub API에서 수집한 원천 활동 데이터를 통합 보관하는 구조. 지표 계산의 입력. |
| Metrics | ActivityData로부터 산출된 4개 핵심 지표(활동성/다양성/협업도/지속성)를 0–100 척도로 표현. |
| AnalysisResult | 분석의 최종 산출물. 종합 점수, 개발자 유형, 강점/약점, 피드백 목록 포함. |
| TrustLevel | 분석 결과 신뢰도. HIGH / PARTIAL / LOW / LIMITED 중 하나. |
| DeveloperType | 점수 기반 개발자 분류. BEGINNER(0–39) / JUNIOR(40–69) / ADVANCED(70–100). |
| RequestStatus | AnalysisRequest의 상태. PENDING / RUNNING / COMPLETED / PARTIAL / CANCELLED / FAILED. |
| Request ID | AnalysisRequest를 식별하는 고유 ID. 진행률 조회, requestId 기준 결과 조회, requestId 기준 PDF 다운로드, 분석 취소에 사용한다. |
| User (인증된 사용자) | GitHub OAuth 2.0으로 인증된 사용자. userId, githubId, sessionId 등 관리. |
| OAuthClient | GitHub OAuth 2.0 API 호출 캡슐화 클래스. Authorization Code + state 검증 플로우 구현. |
| AuthenticationService | 사용자 인증 및 세션 관리 오케스트레이션 클래스. OAuth state 단일 사용, 10분 만료, pending state 500개 제한을 적용한다. |
| GithubApiClient | GitHub REST API 호출을 캡슐화. Rate Limit 및 재시도 로직 내장. |
| MetricCalculator | ActivityData로부터 4개 지표를 산출하는 계산 클래스. |
| CompetencyScorer | Metrics를 종합 점수로 변환 및 Developer Type 분류. |
| FeedbackGenerator | 약점 목록으로부터 맞춤형 FeedbackItem 생성. |
| ReportAssembler | AnalysisResult를 UI/리포트(PDF) 렌더링용 모델(Map)로 변환. |
| ReportGenerator | 리포트 모델을 OpenPDF로 A4 PDF 바이트로 렌더링하고 다운로드 파일명을 생성. |
| AnalysisService | 분석 파이프라인 전체 오케스트레이션. |
| AnalysisRepository | AnalysisRequest와 AnalysisResult 영속화 및 조회, 최근 이력 제한, 만료 데이터 정기 정리를 담당. |
| Cancelled Request | 사용자가 진행 중 분석을 취소하여 RequestStatus.CANCELLED로 종료된 요청. |
| Retention Cleanup | 오래된 요청, 결과, 지표를 스케줄러로 정리하여 저장소 크기와 조회 성능을 관리하는 운영 정책. |
